# Multiagent-Minecraft 项目说明

本文件是本仓库内 AI 编码/研究 Agent 的长期上下文。开始工作前先阅读本文件；进入具体目录后，也应查找并遵守更靠近目标文件的 `AGENTS.md`。

## 项目定位

本仓库基于 MindCraft 二次开发，是一个 Agentic Minecraft 多智能体实验平台。当前核心研究目标不是只提高任务成功率，而是建立一个能够观察、解释并最终辅助管理多智能体团队的系统。

总体研究愿景可概括为“多智能体团队管理系统”，包含三条相关但尚处于不同成熟度的方向：

1. 协作过程评估：突破只看最终分数/成功率的评估方式，记录团队中等待、重复劳动、任务冲突、资源堵塞、依赖、承诺兑现和修复等过程。
2. 宏观决策与现场信号对齐：把底层、快速、多模态的环境信号压缩成高层决策可用的宏观状态；网络重整化是当前探索方向，不要把它描述成已经完成验证的成果。
3. 长期连锁反应预测与规划：在事件早期推演其可能引发的后续依赖和风险，并在连锁反应发生前介入；这同样是研究目标，而非本仓库已经实现的能力。

当前仓库中证据最充分、最应优先推进的是第一条：开放世界中的协作过程表征与评估。

## 当前研究判断

- 最终成功或得分只是过程的投影，不能单独说明协作质量。
- 不要急于把协作压成一个总分。优先构造带时间、证据和置信度的过程表征。
- 当前有用的基本链条是：`承诺候选 -> 可观察行动 -> 世界/库存变化 -> 跨 Agent 依赖 -> 异常 -> 发现/修复 -> 终局`。
- 同一轮实验可以同时包含互补、重复、冲突和修复；这些关系也会随时间改变。
- “聊天多”“命令多”或“Agent 很忙”不等于协作有效。必须检查这些活动是否改变了资源流、世界状态或任务进度。
- 规则适合检出可直接观察的事件；Agentic AI 适合串联上下文、提出解释和表达不确定性。优先采用二者结合的方法。
- 不预先把 Minecraft 的开放任务塞入一套固定人工依赖图。可以使用少量通用概念，但具体协作模式应尽可能从轨迹中发现，并保留 `unknown`。

## 数据范围与入口

主要原始数据位于：

- `experiments/out/trial_101` 至 `experiments/out/trial_300`
- 共 200 轮：100 轮 construction，100 轮 crafting
- 原始轨迹结构见 `experiments/TRACE.md`

每轮可能包括：

- `trace/meta.json`：任务、Agent、场地和时间元数据
- `trace/world_events.jsonl`：位置、方块、聊天、加入/离开等世界观察
- `trace/<agent>.trace.jsonl`：命令、输出、库存和 Agent 侧消息
- `trace/prompts/<agent>/`：完整模型输入输出
- `scores.tsv`：任务进度时间线
- 运行日志、录像和 replay 产物（并非每轮都有）

重要分析材料位于 `experiments/analysis/open_world_coordination/`：

- `COLLABORATION_PROCESS_ANALYSIS.md`：200 轮技术分析与方法边界
- `TRIAL_106_162_CHRONOLOGY.md`：两个案例的可复核时间线
- `TRIAL_106_162_STORY.md`：面向人类的案例叙述范例
- `derived/`：统一事件流、承诺账本、过程片段、指标及质量报告

除非任务明确扩大范围，主分析不要混入 `trial_001-040`、replay、render 或 video 数据。不要把生成/回放事件当成原始实验真值。

## 证据纪律

分析轨迹时始终区分：

- `O（观察事实）`：可由消息、命令、库存、位置、方块、得分或运行日志直接复核。
- `I（合理解释）`：多个观察共同支持的过程解释，但不是 Agent 心理状态的直接记录。
- `H（研究假设）`：需要新实验验证的机制或因果判断。

所有重要判断应尽量给出 `trial_id`、相对时间、源文件和行号/事件标识。不得把时间相近自动写成因果关系，也不得把低置信度的最近位置归因写成责任认定。

特别注意：

- 静止时不产生 pose 事件；最近位置不一定代表动作执行者。
- 同期库存一增一减不自动等于成功交接，优先寻找命令结果和下游消费证据。
- 两人都认领完整任务不自动等于冲突；必须寻找可观察的目标竞争或世界状态损害。
- 正确方块消失不自动等于队友恶意拆除；先描述状态丢失、持续时间和是否恢复。
- `!newAction` 开始执行不自动等于承诺完成。
- Agent trace 报告成功与 runner 的终局标记可能不一致；把它记录为测量/终局检测不一致，而不是直接判定协作失败。
- `unknown` 是合法且必要的结论。证据不足时不要强行分类。

## 推荐分析方式

1. 先建立统一时间线，对齐 Agent 侧行为、世界真值、库存和得分。
2. 找出承诺、请求、交接、依赖、进度平台、状态损失、策略改变和修复等事件。
3. 将事件组织为局部过程片段，不先假设整轮只有一种协作类型。
4. 用可复核规则检测高置信度现象；将模糊样本保留给语义分析或人工审计。
5. 比较“相同结果、不同过程”和“不同结果、相似过程”的案例，避免只围绕成功/失败组织结论。
6. 抽样人工复核，并报告覆盖率、未知比例、误差来源和反例。

做开放编码或探索性分析时，不要先阅读会给现象贴标签的汇总结论，以免把既有分类强加给数据。确需使用已有报告时，要明确这是复核、扩展还是独立分析。

## 写作与讲解方式

用户希望研究结果像人类讲解员所讲的故事，而不是指标堆砌。

- 从一个具体团队、具体时刻或具体矛盾开始，再说明为何需要某项统计。
- 先回答“发生了什么”，再回答“为什么管理者应该关心”，最后说明证据边界。
- 每个数字都应服务于一个问题，例如等待多久、求助是否解除阻塞、工作是否重复、资源是否形成下游进展。
- 不要用消息数、命令数等代理指标直接替代协作质量。
- 案例可以展开，但必须保留资源条件、实验条件和归因覆盖率等公平比较边界。
- 面向非技术读者时少用术语；必须使用时，先用日常语言解释。
- `TRIAL_106_162_STORY.md` 是当前认可的叙事风格参照。

## 实现与验证约定

- 优先复用仓库现有的解析器、事件字段和脚本，不为同一数据另造不兼容格式。
- 原始轨迹视为只读证据；派生结果写入明确的 analysis/derived 目录，并保留生成脚本。
- 对 JSON/JSONL/TSV 使用结构化解析，不用脆弱的文本拼接代替。
- 规则必须说明最小定义、排除条件、证据来源和不确定性。
- 修改分析代码后，先用少量已知 trial 做定点验证，再运行全量 200 轮；全量结果应进行抽样审计。
- 修改运行平台时，遵循根目录 `README.md` 和 `experiments/README.md` 的环境说明；Minecraft 任务依赖特定服务端、端口、权限和清场步骤。
- 不提交或输出 `keys.json` 中的凭据。示例只使用 `keys.example.json` 或占位符。
- 不修改或删除用户已有的实验输出，除非任务明确要求。
