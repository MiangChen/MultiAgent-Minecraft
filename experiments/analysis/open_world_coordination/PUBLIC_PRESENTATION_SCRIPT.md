# 5 分钟口头讲解稿：两个满分团队，合作得一样好吗？

各位先看两个 Minecraft 里的 AI 团队。

它们面对同一类建造任务，最后都拿到 100 分。按照传统评估，这两个团队表现一样好。

但过程完全不同。

在 `trial_106` 中，两名 Agent 各完成约 32 个能够确认的正确方块，几乎没有直接沟通，用了约 295 秒。

在 `trial_162` 中，一名 Agent 完成约 110 个方块，另一名约 18 个。它们交换了 7 条定向消息，完成时间约为 507 秒。

一个更像平衡并行，另一个更像一人主导。最终得分没有告诉我们这些差别。

这正是我们研究的问题：多智能体系统不仅要知道任务有没有完成，还要知道团队是怎样工作的。

如果只看结果，我们看不到一人包办、重复劳动、资源堵塞、互相覆盖、长期等待，也看不到问题出现后有没有被修复。

所以我们没有立即创造一个新的合作总分。我们先建立了一本协作流水账，持续记录五件事：Agent 承诺做什么，实际做了什么，世界和库存怎样变化，谁为谁提供了下一步条件，以及异常发生后有没有恢复。

200 轮实验带来三个核心发现。

第一，很早开始行动，不等于已经形成分工。

100 轮建造中，有 86 轮出现多个 Agent 在早期同时认领整张蓝图。后来有 84 轮表现出行动互补，但这种互补往往与它们各自持有什么材料一致。也就是说，看起来像分工，可能只是库存限制替团队做了决定，并不表明它们真正理解了彼此的角色。

第二，传递资源，不等于合作链已经形成。

100 轮合成中，我们确认了 66 次成功交接，但只有 37 轮能够继续看到接收者用这些资源执行下游工作。管理系统不仅要知道物品从谁手里到了谁手里，还要知道这次交接是否真的解除等待、启动下一步任务。

第三，发现冲突还不够，还要看能否恢复。

建造轨迹中出现了 755 个正确世界状态丢失序列。其中 366 个后来恢复，389 个直到结束仍未恢复。如果只统计破坏次数，我们会把短暂调整和永久损伤混在一起。恢复能力才告诉我们团队能否自行消化异常。

这项工作的价值，是把任务记分牌升级成团队管理仪表盘。它可以帮助我们发现职责重叠、负载集中、资源阻塞、重复劳动和长期未修复的冲突，并为重新分工、资源调度和主动干预提供依据。

我们也保持谨慎。当前实验主要是建造和合成；部分方块行为无法可靠归因；我们建立了过程诊断基础，但还没有证明自动干预一定有效。

我们正在把多智能体评估从结果记分牌，升级为能够看见分工、依赖、冲突和恢复过程的团队管理仪表盘。
