# 两支都拿到 100 分的 AI 团队，真的合作得一样好吗？

我们把同一张 Minecraft 金字塔蓝图交给两支由 Andy 和 Bob 组成的团队。蓝图共有 168 个方块，两支团队最后都完成了建筑，也都拿到了 100 分。

如果只使用传统的任务评估，分析到这里就结束了：

```text
trial_106：100 分，成功
trial_162：100 分，成功
```

但如果我们想真正理解、管理甚至改进一个 AI 团队，这两行结果远远不够。我们还会关心：两个人有没有分工？任务是共同完成的，还是最后由一个人接管？中间有没有长期停滞？求助和回应有没有真的改变行动？已经完成的工作是否被破坏，又是否得到修复？

为了回答这些问题，我们沿着两轮实验的完整记录，把 Agent 的承诺、工具执行、库存变化、世界变化和建筑进度放回同一条时间线上。下面不是对两支团队简单打分，而是尝试讲清楚：两个相同的 100 分，分别是怎样发生的。

> 本文根据[原始过程时间线](TRIAL_106_162_CHRONOLOGY.md)重述。两轮的资源条件不同：`trial_106` 没有资源缺口，`trial_162` 的常用石材存在 33% 缺口。因此，这两个案例适合说明“相同结果可以隐藏不同过程”，不适合直接证明哪支团队更会合作。

---

## 第一支团队：没有开成会，却自然形成了两条工作线

第一支团队来自 `trial_106`。

任务开始时，完成金字塔所需的材料基本齐全，但分散在两个人手里。Andy 拿到的是石头、石砖、安山岩、闪长岩和花岗岩等常用石材；Bob 拿到的是金块、石英、石英柱和萤石等稀有材料。

也就是说，两个人都只能完成建筑的一部分，但他们的材料恰好互补。

### 开始：两个人都认为自己要完成整座建筑

任务开始后的第一秒，Andy 和 Bob 同时收到目标：按照蓝图完成整座建筑。随后，两个人各自检查了蓝图。

12.6 秒时，Bob 率先提出一个完整计划。他不是说“我负责金块和石英”，而是说自己要完成整张蓝图，已有材料就直接使用，缺少的材料再想办法采集或制作。

大约半分钟后，Andy 也提出了自己的长计划。他同样没有把任务限制在某一层、某一区域或某一种材料，而是准备完成整座建筑。

所以，在语言层面，他们其实没有形成明确分工。两个人都认领了同一个完整目标。

系统曾尝试让 Andy 主动与 Bob 建立会话，但由于参数错误，这次沟通没有成功。此后整轮实验也没有出现有效的定向同伴消息，更没有一句类似“你负责稀有材料，我负责石材”的角色确认。

如果我们只看对话记录，很容易得出结论：他们没有商量，也没有分工。

但世界里的行动讲出了另一个故事。

### 第一阶段：库存替他们完成了分工

Bob 从 12.6 秒开始执行第一个长计划。他手里主要是金块、石英和萤石，因此实际工作自然集中在这些稀有材料上。

Andy 在 41.1 秒开始主要建造。他的计划明确要求优先使用手里的各种石材，并跳过自己没有的金块、石英和萤石。于是，虽然他嘴上仍然认领整座建筑，实际执行范围却被库存缩小到了石材部分。

从这一刻起，两条工作线开始重叠推进：Bob 放置稀有材料，Andy 铺设常用石材。两个人没有开会，也没有确认彼此的角色，但资源条件把他们推向了互补的位置。

建筑进度很快开始增长：

- 约 98 秒达到 10%；
- 约 110 秒超过 30%；
- 约 122 秒超过 50%；
- 约 139 秒达到 70%；
- 约 148 秒达到 80%。

这段进度不是一人完成后另一人再接手，而是两条长执行窗口大量重叠。库存快照也与进度同步变化：Andy 的石材连续减少，Bob 的金块、石英和萤石也分阶段消耗。

因此，这支团队确实出现了行动上的互补，但这种互补主要是由初始资源配置塑造出来的，而不是两名 Agent 明确协商出的共同角色。

### 第二阶段：大规模铺设结束，团队开始逐点补缺

155 秒以后，Andy 的第一轮大规模石材铺设基本结束。他重新检查蓝图，开始寻找自己还能补上的精确位置，并在随后约 50 秒里用掉最后一批花岗岩和闪长岩。

与此同时，Bob 继续第二轮长执行，把剩余的金块、石英和石英柱放入建筑。到 266 秒左右，Bob 的稀有材料也基本耗尽，建筑已经接近完成。

269.2 秒，Bob 再次检查蓝图，发现五层金字塔中有三层已经完整，只剩两个普通石材位置：一个石砖和一个石头。

这时，任务已经从“共同铺设整座建筑”转变成“修复两个精确缺口”。Bob 随后承诺只处理最后两个方块，并明确要求不要破坏已经完成的区域。

294.8 秒，建筑达到 100%，任务成功。

整轮过程中，没有观察到已经正确放置的方块后来又消失。因此，这支团队不需要经历“破坏后再恢复”的过程，进度虽然有快慢变化，但总体保持向前。

### 32:32 到底说明了什么？

在能够可靠确认执行者的正确放置中，Andy 完成了 32 个，Bob 也完成了 32 个。这个数字很容易被讲成“两人各做了一半”，但证据并不支持这么强的说法。

本轮只有 33.3% 的蓝图相关世界变化能够满足严格归因条件，另有 128 个事件无法安全判断是谁执行的。因此，32:32 只是**可确认部分的贡献相对平衡**，不是全部劳动的精确统计。

而且，两个人也不是每个阶段都平均出力。Andy 的主要贡献集中在前半段的石材铺设，Bob 则在中后段持续补充稀有材料。最后看起来平衡，是不同阶段的贡献累积出来的。

### 如果把它当作一个团队来管理

我们不能简单地说，这支团队已经学会了高质量协商。它没有建立明确的角色协议，也没有表现出对队友计划的清晰确认。

更准确的评价是：

> 这是一支依靠资源配置实现安静并行的团队。它完成得快，过程稳定，可确认贡献相对平衡；但协调主要来自环境替它们划定了工作边界，而不是它们主动形成了共同分工。

这一区别很重要。因为在资源边界清楚时，这种“自然分开”可能工作得很好；一旦任务开放、资源重叠或依赖变复杂，同样的团队未必还能自动形成有效分工。

## 第二支团队：快速推进之后，停了 247 秒

第二支团队来自 `trial_162`。

他们面对的是同一张 168 方块蓝图，Bob 手里的金块和石英也与第一轮近似，但 Andy 的常用石材少了 33%。这意味着，初始库存只能完成一部分建筑，之后必须额外采集、复用材料，或者改变执行方式。

这个资源缺口在最终的 100 分里完全看不出来，却重塑了整轮协作过程。

### 开始：还是两个人同时认领整项任务

13.1 秒时，Bob 首先表示自己将立即建造整座建筑，并计划在需要时采集缺失材料。

31.6 秒时，Andy 也提出完整建造计划：先使用已有石材，再寻找缺少的金块、石英和萤石。

和第一支团队一样，两个人没有预先划分区域、材料或阶段，而是同时认领了全任务。自动建立会话的尝试也再次因为参数错误失败。

不过，开始阶段的进度并不差。Bob 先使用一部分金块和石英，Andy 随后进入长执行窗口，把手里的常用石材不断转化成建筑：

- 约 83 秒达到 10%；
- 约 94 秒达到 20%；
- 约 106 秒达到 30%；
- 约 116 秒达到 40%；
- 约 130 秒达到 47.62%。

如果只看前两分钟，这支团队和第一支团队很像：两个人都在执行，建筑也在快速增长。

### 第一个转折：材料耗尽，但两个 Agent 仍在不断“做事”

130 秒左右，Andy 的常用石材全部用完，库存里只剩三件工具。此时，建筑还不到一半。

Bob 已经意识到缺少材料，并启动了一个持续约 139 秒的采集和补缺计划。Andy 也提出要从蓝图外收集剩余材料，并执行了约 72 秒。

从 Agent 日志看，两个人都没有停止：他们仍在调用工具、更新计划、检查蓝图并尝试寻找材料。

但从世界结果看，建筑进度一直停在 47.62%。

这揭示了一个传统评估和普通运行日志很容易遗漏的现象：

> Agent 一直在行动，不等于团队一直在前进。

如果只统计命令数量，我们会认为团队非常活跃；如果只统计计划数量，我们会看到它们持续尝试；只有把这些活动与世界变化对齐，才会发现所有努力都没有转化为新的有效成果。

### 第二个转折：求助发生了，但阻塞没有解除

236.7 秒时，Andy 第一次成功向 Bob 发出明确求助：请 Bob 帮忙收集和放置缺少的金块、石英、萤石和石材。

244.3 秒，Bob 回应：“我来处理这些缺失方块。”十秒后，他又表示“快完成了”。

从对话上看，这像是一段标准的协作闭环：

```text
Andy 发现资源缺口
→ 向 Bob 请求帮助
→ Bob 接受任务
→ Andy 等待并准备验收
```

Andy 随后也说会在 Bob 完成后进行检查。

可是，当他真正再次检查蓝图时，大量方块仍然缺失，得分也继续停在 47.62%。语言中的“接手”和“快完成了”，没有得到同期世界进度的支持。

这就是为什么不能把消息数量直接当作合作质量。在这轮实验中一共出现了 7 条定向消息，但消息是否有价值，不能只看它有没有被发送，而要看它是否改变了行动、资源流或任务状态。

Bob 的回应构成了一个可观察的承诺，却没有及时解除团队的实际阻塞。

### 247 秒的平台：忙碌掩盖了停滞

从约 130 秒第一次达到 47.62%，到 377 秒策略真正改变，建筑大约 247 秒没有取得正向进展。

这段平台期不是简单的“两个 Agent 都在发呆”。相反，他们有计划、有执行、有求助、有回应，也多次检查世界状态。问题在于，这些活动没有形成有效的完成路径。

从团队管理角度看，这比单纯空闲更难发现。表面上每个人都很忙，底层工具也持续运行；真正需要被监控的是“活动和成果之间的脱节”。如果系统只能看到 Agent 是否在线、是否调用工具，就会误以为一切正常。

### 第三个转折：Andy 改变了完成任务的方法

292 秒时，Andy 开始提出一个更激进的策略：如果正常放置无法完成，就尝试直接修改世界中的方块。第一次尝试主要在检查可用接口，没有带来得分增长。

377.1 秒，他进一步提出使用精确坐标和 Minecraft 的直接世界/方块放置方式补齐蓝图，并启动了一个持续约 100 秒的长执行计划。

这里必须保留一个关键边界：`trial_162` 的后期成功不能被描述为全部依靠正常生存采集和逐块建造。Andy 的计划明确包含直接 world/block placement fallback，世界日志也记录了与其精确坐标修改相对应的反馈。日志不能证明执行窗里的每一个方块都由同一种接口完成，但能够确认，这次策略升级与后续集中跃升发生在同一时间段。

真正的进度变化出现在执行窗末尾：

- 约 468 秒达到 50%；
- 约 470 秒达到 60%；
- 约 472 秒达到 70%；
- 约 474 秒达到 80%；
- 约 476 秒达到 90%；
- 约 478 秒达到 97.62%。

换句话说，团队此前经历了约四分钟几乎没有成果的采集、计划、求助和检查，最后由 Andy 改变执行方式，在很短时间内集中补齐了大部分剩余结构。

### 成功之前，两个已经正确的方块又消失了

就在策略升级开始后不久，世界中出现了两个异常。

380 秒，一个原本正确的石砖变成了空气；约 390 秒，一个原本正确的石英块也消失了。建筑得分随之发生小幅下降。

基于最近位置的机械归因规则，Bob 是这两个事件的候选者，但置信度只有 0.60。这只能说明 Bob 当时可能是附近的 Agent，不能证明方块由他移除。因此，不能把这两次损失讲成“Bob 拆掉了 Andy 的建筑”。日志缺少足够可靠的执行者真值，责任应当保持未知。

重要的不是急着找责任人，而是看团队如何处理损伤。

这两个正确状态丢失后，分别持续了约 126 秒和 116 秒。直到 480.6 秒，Andy 再次检查蓝图，发现其他层已经基本完成，最后恰好只剩这两个位置。

484.2 秒，Andy 承诺只修复最后两个方块。约 506 秒，世界聊天先后记录了 Andy 的精确坐标修改反馈，紧接着两个位置恢复。消息与方块事件高度对齐，更支持 Andy 发起了最后修复，但由于日志没有独立的命令执行者 ID，表述仍应保留这一链路限制。

507 秒，建筑达到 100%，任务成功。

因此，这轮 100 分背后包含了一条完整的异常链：

```text
正确状态形成
→ 两个正确方块丢失
→ 损伤持续一百多秒
→ 最后被重新识别
→ 精确修复
→ 任务完成
```

最终状态只保留了“建筑完整”，却抹去了它曾经受损、损伤持续多久，以及团队为恢复付出的时间。

### 110:18 是否意味着 Bob 没有工作？

在能够可靠归因的正确放置中，Andy 有 110 个，Bob 有 18 个。这个差距很大，但不能被简化为“Andy 从头包办，Bob 全程没有贡献”。

首先，本轮方块归因覆盖率是 66.3%，仍有约三分之一的蓝图变化不能确定执行者，因此 110:18 同样只是可确认的子集，不是全部劳动的精确总数。

其次，Andy 的 110 个可确认放置中，有 70 个集中在 420 至 480 秒，也就是策略升级后的最后阶段。Bob 在前期确实参与了建设，贡献差距主要是后期任务转为 Andy 主导造成的。

所以更准确的过程不是“一开始就由 Andy 单人完成”，而是：

> 前期两人共同推进；资源耗尽后双双陷入阻塞；显式求助没有及时恢复进度；后期 Andy 通过直接世界/方块放置 fallback 集中接管剩余工作，并完成最终修复。

### 如果把它当作一个团队来管理

这支团队的问题不只是“完成得慢”。更值得关注的是，它在资源耗尽后缺少及时识别和解除阻塞的机制。

它有沟通，但沟通没有立刻转化为结果；它有大量执行活动，但活动长期没有推动世界状态；最终任务成功，却依赖一次较晚的执行策略升级和明显的一人主导。

因此，管理判断应该是：团队前期可以并行工作，但面对资源缺口时缺少可靠的依赖处理和升级机制。系统如果能在进度停止后较早识别“资源已经耗尽、计划反复执行但没有产出、求助承诺没有落地”，就可能不必等待 247 秒才改变策略。

## 把两个 100 分放在一起

两支团队最终都完成了同一张蓝图，但它们走过的是两条不同路线。

| 管理者关心的问题 | `trial_106` | `trial_162` |
|---|---|---|
| 资源条件 | 0% 缺口，材料按类型分开 | 33% 缺口，常用石材不足 |
| 是否形成明确语言分工 | 没有 | 没有 |
| 实际工作如何组织 | 库存塑造出两条互补工作线 | 前期共同推进，后期 Andy 主导 |
| 定向消息 | 0 条 | 7 条 |
| 消息是否解除阻塞 | 不适用 | 求助与回应后，进度没有立即恢复 |
| 主要进度形态 | 大体连续增长，后期逐点补缺 | 约 247 秒无正向进展，之后集中跃升 |
| 可确认正确放置 | Andy 32 / Bob 32 | Andy 110 / Bob 18 |
| 方块归因覆盖率 | 33.3% | 66.3% |
| 正确状态丢失 | 未观察到 | 2 个，后来全部恢复 |
| 后期执行方式 | 常规建造与补缺 | 包含直接 world/block placement fallback |
| 完成时间 | 约 295 秒 | 约 507 秒 |
| 最终得分 | 100 | 100 |

这里不能忽略资源条件：`trial_106` 没有缺口，`trial_162` 有 33% 缺口。第二轮更慢、更曲折，不能全部归因于协作能力；同样，第一轮更平稳，也可能主要受益于资源已经被预先合理分配。

这两个案例真正支持的结论不是“零沟通比七条消息更好”，也不是“第一支团队一定更聪明”。它们支持的是一个更基础、也更重要的判断：

> 最终得分把团队过程压缩得太厉害了。

一个 100 分背后，可能是资源边界清楚、两条工作线安静并行；另一个 100 分背后，可能是资源耗尽、长期阻塞、承诺未落地、执行方式升级、一人接管和异常恢复。

## 为什么需要协作过程评估？

传统任务评估回答的是：“事情最后做成了吗？”

协作过程评估还希望回答：

- 团队是怎样做成的？
- 分工是主动形成的，还是环境碰巧替它们分好了？
- Agent 看起来很忙时，世界是否真的在前进？
- 求助和承诺是否转化成了行动与成果？
- 谁的工作成为了队友下一步的条件？
- 正确状态被破坏后，问题持续了多久？
- 最终成功是否依赖某一个 Agent 在后期接管？

这些问题不是为了再给团队合成一个抽象总分，而是为了让管理者知道什么时候应该介入，以及介入什么。

如果系统能看见 `trial_106` 的资源互补，就可以把这种隐式分工显式化，让 Agent 更早知道彼此的职责；如果系统能看见 `trial_162` 中“持续执行但 247 秒没有进展”，就可以更早触发资源检查、任务重分配或策略升级；如果系统能识别正确状态丢失及其持续时间，就可以区分短暂调整和真正的未修复损伤。

因此，这两个案例所展示的，不只是更详细的实验日志，而是一种团队管理视角：

> 不只记录任务有没有完成，还要看见分工如何形成、依赖在哪里阻塞、承诺是否兑现、异常是否恢复，以及成功过程能否稳定复现。

完整事件、证据行和归因规则可在[原始过程时间线](TRIAL_106_162_CHRONOLOGY.md)中复核；技术背景与整体统计见[协作过程分析报告](COLLABORATION_PROCESS_ANALYSIS.md)。
