# Multiagent-Minecraft 200 轮开放世界协作过程分析

> 分析对象：`trial_101`–`trial_300`。分析前未读取 `experiments/RESULTS.md`；开放编码和全量分类完成后才读取，用于末尾对照。主分析未纳入 `trial_001–040`、replay、render 或 video。生成日期：2026-07-20。

## 摘要

这 200 轮轨迹不支持用单一 cooperation score 描述协作。更稳定的表征单位是一个带证据和置信度的过程链：**承诺候选 → 可观察行动 → 世界/库存变化 → 跨 Agent 依赖 → 异常 → 发现/修复 → 终局**。同一轮可以同时呈现互补、重复和损害，且这些关系会随时间改变。

本分析对齐了 999,196 条事件，提取 1,679 个承诺候选，划分 5,007 个事件驱动片段。建造中最普遍的不是显式协商，而是“多个 Agent 同时声称完成整张蓝图，随后被初始库存推向不同材料的隐式分工”：86/100 轮出现全任务承诺重叠，84/100 轮达到严格的库存-放置对齐阈值，65/100 轮在至多两条定向同伴消息下仍出现行动互补。合成中则能看到强得多的显式依赖：共确认 66 次 `!givePlayer` 交接，37 轮形成“交接→接收者合成”的完整链。

异常不是单一类型。建造中 67/100 轮至少丢失过一个已正确方块；共有 755 个“正确状态丢失”序列，其中 366 个随后恢复，389 个到轨迹结束仍未恢复。合成中可见交接空间条件失败后重试、执行失败后的角色反转、单边反复求援、以及错误地转向环境搜索。另有 6 轮命令轨迹明确报告目标已合成，但运行器最终没有成功标记，说明终局判定本身也会扭曲协作评价。

以下标记贯穿全文：

- **观察事实 O**：可由消息、命令、库存、姿态、方块或得分事件直接复核。
- **合理解释 I**：由多源证据共同支持，但不是 Agent 心理状态的直接记录。
- **研究假设 H**：可用于后续实验，不在当前数据中作因果断言。

## 2. 方法

### 2.1 承诺候选

承诺定义为“某 Agent 声称将由谁改变什么可观察状态”。普通问候、状态陈述和纯检查命令不算承诺。四个来源分开保存：

| 类型 | n | 可观察跟进 | 严格完成 | 说明 |
|---|---:|---:|---:|---|
| assigned task goal | 442 | 91.0% | 47.5% | 系统目标，不等同于 Agent 自主分工 |
| self action commitment | 648 | 100% | 55.7% | `!newAction` 进入执行窗口即算跟进，完成要求更强证据 |
| dialogue commitment | 487 | 56.3% | 20.5% | 明确 “I'll / I will / crafting now” 等前瞻声明 |
| requested commitment | 102 | 37.3% | 23.5% | 明确请求另一 Agent 提供行动或资源 |

全部候选中 695 条完成，592 条被后续计划修改或替代，392 条保持未知；1,362/1,679 有可观察跟进。`!newAction` 的“开始执行”不能自动推出目标完成，因此账本同时保留 follow-through 与 status。

承诺间关系在 45 秒并发窗口内编码为：目标/蓝图范围重叠、潜在互补或请求依赖。关系是对称记录，故 1,822 条 overlap、132 条 potentially complementary、324 条 dependency 不是独立 pair 数量，也不是得分。

### 2.2 协作片段

片段边界来自承诺、交接、得分改变或 60 秒平台、正确状态丢失、帮助/等待/约束消息、离开和终局。没有使用固定窗口。每个片段聚合进入时承诺、命令、消息、库存变化、方块变化、交接候选和得分变化。

片段层面只给 8 种可局部定位标签；3,735/5,007 片段保留 `unknown`，这是有意的高精度策略，而不是把缺少证据解释成“没有协作”。低置信度库存协变单列为 `ambiguous_inventory_covariation`，不计入确认交接。

### 2.3 开放编码与全量应用

固定随机种子 `20260720`，每个实验组等概率抽 3 轮，共 27 轮：105/112/119、130/133/140、143/151/159、164/167/176、183/185/191、201/210/212、234/243/247、255/259/263、276/287/300。编码时没有读取 `RESULTS.md`，也没有预设“互拆、停滞、重复劳动”等类别。

候选类别先回答最小定义、支持证据、排除边界、现象层级和成败独立性，再写成保守规则应用到 200 轮。规则输出不覆盖原始观察，所有标签都有置信度和证据位置。

## 3. 八个主要过程发现

### 3.1 承诺通常很早形成，但“早”不等于“分工已形成”

**O** 建造 100/100 轮都有实质承诺，首个非系统承诺中位数 13.85 秒（四分位 12.94–22.90 秒）；合成 88/100 轮可识别到非系统承诺，中位数 13.56 秒。建造 86/100 轮在 45 秒内有两个以上 Agent 都认领完整蓝图。

**I** 早期语言更常是并行的全任务认领，而不是互斥的角色协议。`trial_105` 中 Andy 和 Bob 在 13.1/14.2 秒都声称将完成整张蓝图（`trace/andy.trace.jsonl:22`; `trace/bob.trace.jsonl:17`），但后续物料和放置仍高度互补。

**不能推出**：重叠承诺不等于冲突意图；很多执行函数在目标已正确时会跳过，两个重叠计划可在世界层面和平共存。

### 3.2 建造协作主要由库存约束“塑形”，不是由聊天协商产生

**O** 84/100 建造轮达到库存-放置对齐阈值；65/100 同时满足多 Agent 实质贡献且定向同伴消息不超过 2 条。建造总定向消息 226 条，合成总定向消息 748 条。`trial_106` 零定向消息，但保守归因的正确放置是 Andy 32、Bob 32（`world_events.jsonl:781,803,815,823,852`）。

**I** 初始材料拆分是一个外部协调装置：即便语言都认领全任务，实际行动仍会被各自持有的方块类型分开。这是一种“隐式互补”，与有意识商定角色不同。

**反例**：`trial_164` 同样库存对齐 98.5% 且零定向消息，却由 Andy 占 56/65 个可归因正确放置并失败于 49.4%；隐式互补不保证平衡或完成。

### 3.3 显式资源交接在合成中形成可观察的依赖链

**O** 严格确认 66 次成功 `!givePlayer`：H1 18、H2 17、E1 0、E2 31。37 轮满足“成功交接→接收者尝试/完成目标”的定义。`trial_201` 中 Andy 32.7 秒请求 2 根木棍，Bob 51.5 秒命令结果报告 Andy 收到木棍，Andy 64.3 秒发起木镐合成，运行器 68.1 秒标记成功（`andy:14,16`; `bob:11`）。

**I** 这里的合作不是聊天量，而是一个有方向的资源前置条件被执行并被下游行动消费。

**边界**：103 个同期库存增减只有协变证据，没有成功 give 命令，保留为低置信度，不称为交接。

### 3.4 协作恢复常表现为角色或资源方向改变

**O** 3 轮出现“交接失败后成功重试”，7 轮出现“原执行者合成失败→资源转给队友→队友完成”。`trial_243` 首次给铁因 “too close” 失败（`andy.trace.jsonl:22`），Andy 先退开，再于 133.6 秒成功给 Bob，两秒后库存确认，Bob 144.9 秒完成剪刀（`andy:27`; `bob:29,31`）。`trial_287` 中陶瓦先 Bob→Andy，Andy 多次合成报 missing ingredient，随后陶瓦与染料 Andy→Bob，Bob 103.7 秒完成（`andy:7,19,24,27`; `bob:22`）。

**I** 修复不是简单“再试一次”：`trial_243` 改变了空间前置条件；`trial_287` 改变了执行者和资源方向。

### 3.5 正确世界状态的丢失需要与恢复共同描述

**O** 67/100 建造轮至少出现一次正确方块被移除/覆盖；755 个丢失序列中 366 个恢复，389 个未恢复。0% 缺口组 9 轮发生丢失且 9 轮存在至少一次恢复；3-Agent 22% 缺口组 14 轮发生丢失，只有 4 轮出现恢复，14 轮仍有未恢复位置。

**I** 只计“破坏次数”会混合短暂修正、材料重用、路径动作和持久损伤。恢复延迟与是否最终恢复是不同的过程维度。

**归因限制**：平均只有 48.1% 的任务相关方块事件满足严格 Agent 归因条件。因此主结论使用 agent-agnostic 的世界状态序列；只有距离和命令窗口都支持时才描述执行者。

### 3.6 阻塞后会出现计划重述，但语言活动不一定产生新世界效应

**O** 14 轮满足“得分至少 120 秒不变且承诺明确转向缺料/采集”，6 轮进一步出现两个以上新物料计划但仍无得分增长。`trial_158` 最后 397 秒得分停在 73.81%，有 36 条定向消息、36 个承诺、三次后期物料计划和三次未兑现的 Bob→Andy 请求（`scores.tsv:2151`; `andy:35,37,41`; `bob:17,44,54`）。

**I** 这类过程不是“没有思考”，而是承诺更新与可执行前置条件脱节。把所有聊天计作合作会高估它。

### 3.7 同一终局可以由完全不同的过程产生

**O** `trial_106` 与 `trial_162` 都建造成功且终分 100。前者保守归因 32:32、零定向消息、294.8 秒；后者 110:18、7 条定向消息、507.0 秒。`trial_201` 与 `trial_263` 都合成成功：前者含明确请求和木棍交接，后者由 Andy 在轨迹结束前自行制作工作台并直接合成，未观察到输入交接。

**O** 同样失败也可不同：`trial_212` 是 Bob 单边多次求援并搜索不可达木材；`trial_247` 是 Bob 在 Andy 初始持有铁锭的情况下，自行远行搭建从木头到采铁的长链，最终未完成剪刀。

**I** 终局是过程的投影，不是过程本身。

### 3.8 任务成功判定也可能与过程证据不一致

**O** `trial_208/255/262/267/270/297` 的 Agent trace 至少一次明确报告成功合成目标，但 run log 没有成功标记。`trial_255` 中 Andy 和 Bob 都报告木镐合成成功，且 Andy 放置共享工作台供 Bob 使用（`andy:9,14`; `bob:14`），运行器仍在 600 秒超时。

**I** 这不是“协作失败”的同义词，而是终局检测、物品保持、任务轮询或关闭时序中的至少一种不一致。

**不能推出**：命令结果一定等于最终持久世界状态；目标 inventory 快照常因任务立即结束而缺失。

## 4. 自下而上的协作过程词典

| 现象 | 最小定义与证据 | 不应算入 | 层级 | 与成败独立性 / 典型案例 |
|---|---|---|---|---|
| whole-task commitment overlap | 两个以上 Agent 45 秒内都明确认领完整蓝图 | 只有共同系统 `!goal`；仅共享任务名 | 语言/跨 Agent | 成功 `105/106`，失败 `140/164` |
| inventory-induced implicit specialization | 两个以上 Agent 实质放置，至少 75% 可归因材料符合其初始分配 | 只有一人行动；归因不足；显式分工但无世界变化 | 行动/资源关系 | 84/100 建造轮 |
| balanced parallel execution | 至少两人各有 ≥5 个正确放置，最大份额 ≤68% | 大量未归因时声称精确劳动比例 | 行动关系 | 成功 `106`，失败 `183` |
| quiet action complementarity | 多人材料互补，定向同伴消息 ≤2 | 无消息且只有一人行动 | 行动/关系 | `106/112/183` |
| single-agent execution dominance | 一人占 ≥80% 可归因正确建造动作 | 把未知归因分配给最近者 | 行动关系 | 成功 `162`，失败 `164` |
| verified handoff to completion | 成功 give 命令命名物品/接收者，后者随后执行目标 | 同期库存协变；只说“给了” | 资源/依赖 | `201/210/234/243` |
| shared infrastructure support | 一人放置工作台，另一人随后使用/面向目标合成 | 工作台仅供自己使用 | 世界/依赖 | `255/271` |
| handoff failure then repair | 失败 give 后改变条件并确认成功重试 | 重复说“再给一次”但无成功结果 | 异常/修复 | `243/279` |
| role reassignment after execution failure | A 合成失败，输入转给 B，B 完成 | 仅两人都尝试；无资源方向变化 | 异常/资源/修复 | `287/282/291/295` |
| unanswered request loop | 同一 speaker 对同一队友 ≥3 次请求且无确认交接 | 一次请求；队友明确拒绝后停止 | 语言/依赖阻塞 | `158/212/230` |
| resource search displacement | 失败轮中 ≥5 次环境搜索/采集，且没有确认交接 | 合理搜索后成功；队友本来没有资源 | 行动/资源阻塞 | `212/247` |
| late dependency disclosure | 首次或重复依赖请求发生在 65% 轮长以后 | 早期正常协商 | 语言/时序 | 24 轮；成功反例 `201` |
| correct world state loss | 已匹配蓝图的位置后来变为不匹配 | 清理原本错误的方块；环境远处变化 | 世界关系 | 67 建造轮 |
| correct state restoration | 丢失坐标后来恢复预期方块 | 另一个坐标的进度增长 | 世界/修复 | `112/140/162` |
| unrepaired correct state loss | 丢失位置到结束仍未恢复 | 因 trace 截断无法看见后续时标高置信完成 | 世界/异常 | `140/183/185` |
| resource-blocked progress plateau | ≥120 秒无得分变化，期间承诺明确围绕缺料/采集 | 无日志的静止；完成后等待关闭 | 资源/时序 | 14 轮，`158/190` |
| plan restatement without world effect | 最后得分变化后至少两个新物料计划且无后续得分增长 | 计划变化后确有其他不可评分成果 | 语言→行动缺口 | 6 轮 |
| self-contained task capture | 单一 Agent 成功合成，未观察到输入交接或队友工作台支持 | 终端命令因关机未记录的未知轮 | 行动关系 | H1 7、E1 9、E2 10 轮（严格下界） |
| parallel target overproduction | 两个 Agent 都有成功目标合成结果 | 只有尝试；同一人重复合成 | 行动冗余 | `255/258/271` |
| terminal detector process mismatch | trace 报目标合成成功，run log 无成功标记 | 只有 LLM 声称成功；命令实际报错 | 测量异常 | 6 轮 |
| unknown | 当前规则没有足够证据 | 不把 unknown 当成“无合作” | 不确定性 | 27 轮 trial，3,735 个 episode |

“冲突”在本词典中只用于可观察目标/世界状态竞争，不把两个全任务声明自动标为冲突。没有足够证据证明某 Agent 想伤害队友；只能说某正确状态在某行动窗口内丢失。

## 6. 相同结果、不同过程的匹配案例

| 匹配 | 相同点 | 过程差异 | 研究用途 |
|---|---|---|---|
| `106` vs `162` | 建造成功，终分 100 | 32:32 平衡、0 消息、295s vs 110:18 主导、7 消息、507s | 成功不代表分工相同 |
| `105` vs `112` | 建造成功，终分 100 | 80% 动作集中且无状态丢失 vs 较平衡且出现丢失后恢复 | 相同结果，不同冲突/修复史 |
| `140` vs `185` | 都失败且终分 87.5 | 双 Agent 零定向消息、7 丢失/2 恢复 vs 三 Agent 16 消息、3 丢失/1 恢复 | 同分不等于同交互密度 |
| `143` vs `164` | 都失败、缺料条件 | 67.3 分且后期求援、丢失后恢复 vs 49.4 分、单人主导、没有状态丢失 | 失败机制不同 |
| `201` vs `263` | 木镐成功 | 请求→木棍交接→合成 vs 单人快速闭环 | 显式依赖 vs 资源冗余/自给路径 |
| `212` vs `247` | 600s 左右失败 | 单边求援+不可达木材搜索 vs 队友持铁时仍自行构建采矿技术链 | 沟通断路 vs 依赖识别失败 |
| `243` vs `255` | 局部协作证据强 | 交接失败经退开修复后被终局承认 vs 共享工作台且双方合成，却终局超时 | 修复质量与测量异常分离 |
| `210` vs `158` | 都有明确同伴依赖语言 | 少量消息、一次交接、42s 成功 vs 36 消息、请求循环、397s 平台、失败 | 聊天量不是有效协作代理 |

## 7. Trial story cards

以下每张卡都按“承诺→行动→世界/资源影响→结果”组织；引号只摘取短证据。

### 7.1 `trial_105`：成功但执行集中

- **O 13–14s 承诺**：两人都说要完整建蓝图；Andy “I’ll build it now”（`andy:6,22`），Bob 声称采集石材并建完整蓝图（`bob:6,17`）。

- **I**：语言范围重复，世界行动互补但明显集中；“成功”同时容纳低显式分工和单人主导。

### 7.2 `trial_106`：极少沟通的平衡并行

- **O 12.6/41.1s 承诺**：Bob 认领完整蓝图；Andy 明确只放库存中的石材并跳过稀有块（`bob:14`; `andy:25`）。

- **O 结果**：294.8s，100 分成功。
- **I**：这是“沟通很少但行动互补”的强反例。

### 7.3 `trial_112`：成功前的短暂状态丢失和恢复

### 7.4 `trial_140`：高分、静默互补与未完成修复并存

- **O 12–13s**：两人都认领完整蓝图（`bob:11`; `andy:20`）。

- **I**：同一轮同时有隐式分工、世界冲突和部分恢复，不能压成单标签。

### 7.5 `trial_143`：缺料后晚期披露依赖

- **O 14.6/31.5s**：Bob 认领全任务；Andy 明确只用现有石材（`bob:10`; `andy:26`）。

- **O 565.3s**：Andy 才向 Bob 请求材料/帮助（`andy:60`）。
- **O 结果**：721.4s，67.26 分失败。
- **I**：局部合作良好，但依赖披露太晚且未带来终局完成。

### 7.6 `trial_158`：沟通密集但世界进度停止

### 7.7 `trial_162`：成功的单人主导路径

- **O 结果**：507.0s，100 分成功。
- **I**：与 `106` 相同结果，但不是同一种协作组织。

### 7.8 `trial_164`：隐式分工存在，负载仍失衡

### 7.9 `trial_183`：三 Agent 的全任务重叠与局部互补

### 7.10 `trial_185`：高分失败中的多方实质贡献

- **O 承诺**：Candy 15.3s 认领全蓝图（`candy:14`），其余 Agent 随后也行动。

- **O 结果**：87.5 分失败。
- **I**：这是“失败但局部合作良好”的案例；高参与不等于最终完整。

### 7.11 `trial_201`：请求、交接和下游消费

### 7.13 `trial_212`：单边请求和错误的环境替代路径

## 9. 研究假设和最小验证实验

### H1：结构化角色承诺可能减少全任务认领重叠，但不保证更高完成率

- **数据证据**：86/100 建造轮有全任务承诺重叠；`106` 与 `162` 都成功但负载分布不同。
- **反例**：`105/106` 在重叠承诺下仍成功；重叠本身不是失败充分条件。
- **替代解释**：执行函数跳过正确坐标，使语言重叠未转化为物理竞争；库存拆分已足够塑形。
- **新增字段**：结构化 `goal_id/role/target_coordinates/material_budget/valid_until`。
- **最小实验**：同一库存和蓝图随机分配“自由全任务承诺”与“先发布互斥区域/材料承诺”，比较目标重叠、改计划次数、恢复成本；不以成功率为唯一因变量。

### H2：交接修复取决于 Agent 是否获得可操作的失败语义

- **数据证据**：`243` 的 “too close” 后出现退开和成功重试；3 轮达到严格修复定义。
- **反例**：`201/210` 第一次交接即成功，无需诊断。
- **替代解释**：成功可能只是随机路径变化，而非理解错误信息。
- **新增字段**：give 发起/投掷/物品实体/拾取/接收的统一 transaction_id、双方距离、地形可见性。
- **最小实验**：固定物品与 Agent，在 1/3/8 格和有/无遮挡条件重复交接；随机提供精确错误、模糊错误或无错误消息。

### H3：恢复率和恢复延迟比破坏次数更能区分瞬态协调异常与持久损伤

- **数据证据**：755 个丢失序列中 366 恢复、389 未恢复；`112` 恢复后成功，`140` 部分恢复仍失败。
- **反例**：`164` 失败但没有正确状态丢失。
- **替代解释**：一些丢失可能是局部纠错/支撑操作，不是 Agent 间异常。
- **新增字段**：block action_id、执行者、上层规划目标、是否目标坐标、预期临时性。
- **最小实验**：对同一缺口条件随机提供“你刚移除了正确块”的反馈，测恢复概率/延迟和后续承诺修改。

### H4：资源冗余把失效表型从交接困难转向单人接管或重复产出

- **数据证据**：H 组有 35 次确认交接，E1 没有确认交接且严格观察到 9 个单人闭环；`255/258/271` 有双方成功合成。
- **反例**：E2 有 31 次交接和 7 次角色反转，冗余条件下仍会形成协作。
- **替代解释**：E2 的 recipe/API 异常推动了转交，而不是 Agent 自发分工。
- **新增字段**：目标第一次出现时间、每次额外目标的生产者与资源成本、任务 checker 每 tick 判断。
- **最小实验**：同一配方设置“至少一个”“恰好一个”“每人一个”三个目标，比较单人接管、共享设施和冗余产出。

### H5：执行失败可触发适应性的角色反转

- **数据证据**：E2 中 7 轮符合“失败执行者转资源给队友，队友完成”；`287` 证据链完整。
- **反例**：`212/247` 面对阻塞时没有形成有效角色反转。
- **替代解释**：Agent 只是穷举另一个执行者，而非形成稳定角色模型。
- **新增字段**：明确的 capability belief、error classification、role revision reason。
- **最小实验**：随机向一个 Agent 注入一次可恢复 recipe 失败，操纵是否展示队友能力与库存，观察资源方向和角色声明。

### H6：晚期依赖披露和计划重述可能标记“已识别但未解决”的阻塞

- **数据证据**：24 轮有晚期依赖披露，14 轮有资源平台，6 轮在平台后多次重述计划；`158` 同时具备三者。
- **反例**：`201` 的请求发生较晚但迅速成功。
- **替代解释**：长任务天然给更多机会产生晚消息；平台可能来自执行 API 而非资源。
- **新增字段**：每个承诺的 precondition、当前 blocker、重试次数、失败码。
- **最小实验**：在同一资源缺口下随机共享队友库存摘要或隐藏，比较首次请求时间、请求兑现率和计划重述数。

### H7：终局检测错误会系统性误分类“成功但低效/协作良好”的过程

- **数据证据**：6 轮成功 craft 命令与 run 终局不一致；`255` 同时有共享设施、双方产出和超时。
- **反例**：其余大多数目标 craft 被成功识别。
- **替代解释**：命令的瞬时成功对象后来被丢弃/消耗，终局按持久 inventory 判定可能合理。
- **新增字段**：checker 读取的 Agent inventory、目标实体 UUID、检测结果和失败理由。
- **最小实验**：对这 6 轮回放命令/库存序列，在每次 craft 后强制 checker 快照，区分写入时序、物品丢失和 checker 逻辑。

### H8：静默互补可能由环境约束而非共享理解产生

- **数据证据**：65/100 建造轮满足 quiet action complementarity；`106` 是成功强例。
- **反例**：`164/183` 同样静默互补却失败。
- **替代解释**：材料互补使动作天然分开，Agent 不需要理解队友。
- **新增字段**：Agent 对队友角色/库存的显式 belief、是否读取到队友消息、共同计划版本。
- **最小实验**：保持物料拆分，随机交换库存可见性与通信通道；测行动互补是否在不了解队友时仍同样出现。

## 10. 质量审计与缺失

- 固定种子 `20260721` 分层随机抽查 40/200 轮（建造每组 4，合成每组 5），核对承诺证据行、确认交接的 give 依据、标签证据非空和开放编码边界；40 轮均通过结构与证据检查。审计样本仍有 5 个 `unknown`，这是正确保留不确定性，不是失败项。
- 200/200 有 world events、至少两个 Agent trace、prompt 目录和 run log；JSONL parse error 为 0。
- 100 个合成任务没有 `scores.tsv` 内容，这是任务设计差异，不当作日志缺失；建造 100 轮都有得分时间线。
- 43/200 轮 Agent trace 出现多于 Agent 数的 `trace_start`，表示至少一次重启/重连。统一时间线保留所有段，不把重启后的 Agent 当新人。
- `meta.start_epoch_ms` 全部位数异常；报告的相对时间基于 trace，不能用 meta 原值跨系统对时。
- inventory 每 5 秒变更轮询，快速 craft 后终止时常没有目标物品快照；因此成功合成优先用 cmd result + run log，多源缺失时保留 unknown。
- 世界 observer 记录视距内自然火、水、植物等变化。统一流保留它们，但蓝图语义分析只使用任务坐标/包围盒，避免环境噪声。
- 近距离不是确定执行者。所有 agent-specific 方块结论都受 48.1% 平均归因覆盖限制；agent-agnostic 状态丢失/恢复更稳健。

## 11. 与 `experiments/RESULTS.md` 的事后对照

### 11.1 独立重新发现

1. **成功率和建造终分完全复现**：建造完成 16/1/2/1/1，最终均分 98.25/82.50/70.27/58.63/69.52；合成完成 21/17/21/24。
2. **资源压力下正确状态丢失增加但不是单调**：本分析用更宽的 agent-agnostic 口径看到 9/17/15/12/14 个丢失轮，形状与现有“跨 Agent 拆除”倒 U 描述相容，但不可直接等同。
3. **任务成功与协作过程解耦**：成功建造既可 32:32 也可 110:18；成功合成既可交接也可单人闭环。与现有“高分掩盖低协作”的方向一致。
4. **对等合成容易单人闭环或冗余**：`259` 是成功但单人重复生产；`255` 是共享工作台后双方都生产。
5. **三 Agent 条件有更多交叠和状态损失**：本分析看到 balanced execution 18/20，但 300 个丢失序列中只有 82 恢复，补充了“更多相互依赖也更多破坏”的过程视角。

### 11.2 现有指标过于粗糙之处

- “互拆事件”给出频率和执行者关系，但把恢复前的短暂丢失与到终局仍未修复的损伤放在一起。本分析增加 `loss→restoration/unrepaired` 状态机。
- “停滞”没有区分资源平台、单边求援、错误的环境搜索、命令 API 失败或 Agent 重启。本分析用承诺、失败码、资源方向和世界变化拆开。
- “一人包办/一方零产出”不能表示共享工作台、先提供资源再由一人合成、或执行失败后的角色反转。
- “重复合成”没有区分跨 Agent 冗余与同一 Agent 因 inventory/checker 不一致反复合成；`259` 是后者。
- 只按距离归因方块会隐藏不确定性。本分析显式报告候选距离、姿态年龄、活动命令窗和 `agent=null`。

### 11.3 新发现或此前未单独统计

- 86% 建造轮的**全任务承诺重叠**与 84% 的**库存诱导隐式分工**同时存在。
- 65% 建造轮的**静默行动互补**：很少定向消息但材料动作互补。
- **恢复过程**：366 个正确状态丢失序列被恢复，389 个未恢复。
- **交接修复的空间机制**（too close→退开→成功）。
- **执行失败后的角色反转**（7 轮）。
- **晚期依赖披露、请求循环和平台期计划重述**。
- **终局检测与命令过程不一致**（6 轮）。

### 11.4 被原始过程反例挑战的结论

`RESULTS.md` 将 H1 描述为“材料互补，完成必须经 `!givePlayer`”。任务文件给 Andy 10 个木板、Bob 10 根木棍，但 10 个木板足以由 Andy 自行做工作台、把木板做成木棍并制作木镐；严格轨迹中至少 7 个 H1 成功轮由单一 Agent 完成且没有确认交接。因此 H1 是**初始资源分置**，但不是**状态空间上强制交接**。这也解释了现有表中 H1 同时出现大量“一人包办”。后续若要验证强制依赖，应让任一单方库存都无法沿替代配方闭环，并封闭环境采集旁路。

另一个挑战是把 run success 当作过程真值。6 个反例表明“目标曾被成功合成”与“终局成功标记”不是完全一致；任何用成功标签训练 cooperation evaluator 的工作都应保留 checker 决策日志。

## 12. 结论：建议的任务无关表征

一个可迁移到其他开放世界任务的协作表征至少需要以下对象，而不是预列 Minecraft 行为类别：

1. **Commitment**：actor、issuer、目标状态、目标对象/区域、预期输出、依赖、接收者、修订和置信度。
2. **Action window**：命令、执行时段、结果、错误语义、所处位置。
3. **State transition**：世界/库存从什么变为什么，是否命中目标，执行者归因及其不确定性。
4. **Dependency edge**：谁提供了谁后续行动的前置条件；资源、设施、空间支撑和信息应分开。
5. **Coordination episode**：由承诺/资源/状态边界触发，允许多个现象共存。
6. **Anomaly and repair**：异常出现、持续、被谁发现、策略如何修改、状态是否恢复及恢复延迟。
7. **Outcome layer**：任务终局、局部成果和 checker 可靠性独立保存。

这套表示不会把成功当作合作、不会把聊天当作承诺、不会从方块变化虚构意图，并能描述建造、合成及未来未见过的任务。它把研究问题从“Agent 好不好”改写为“哪些声明被执行、哪些依赖被满足、哪些状态被改变、异常如何被识别和修复”。
