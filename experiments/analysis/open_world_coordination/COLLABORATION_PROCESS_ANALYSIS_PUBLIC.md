# 两个 AI 团队都拿到满分，它们真的合作得一样好吗？

> 基于 200 轮 Minecraft 多智能体实验的协作过程讲解版。技术细节、原始证据位置和完整分析见[源报告](COLLABORATION_PROCESS_ANALYSIS.md)。

## 先看两个满分团队

两个 AI 团队面对同一类建造任务，最后都拿到了 100 分。按照传统评估，它们没有区别：建筑完成，任务成功。

但过程记录讲出了两个不同的故事。

在 `trial_106` 中，两名 Agent 各完成约 32 个能够可靠归因的正确方块。它们没有发送定向消息，在约 295 秒内完成了建筑。

在 `trial_162` 中，一名 Agent 完成约 110 个方块，另一名约 18 个。它们有 7 条定向消息，完成时间约为 507 秒。

前一个团队更接近平衡并行，后一个更像由一名 Agent 主导。两轮都是满分，但工作分配、沟通成本和完成速度明显不同。[配对图](figures/matched_process_pairs.png)展示了这种“结果相同、过程不同”的现象。

这带来一个简单但重要的问题：**两个团队得分相同，它们真的合作得一样好吗？**

## 为什么只看任务结果不够

最终得分只告诉我们世界最后变成了什么样，却看不到团队是怎样走到终点的。

一个任务可能由一名 Agent 包办，其他成员几乎没有贡献；也可能由多人合理并行。团队可能重复制造同一个物品，也可能让关键资源长期停在某个成员手里。一个 Agent 还可能拆掉已经正确的工作，或者在队友等待时独自走一条漫长但无关的路线。

这些差别会影响效率、稳定性和扩展能力，却可能得到完全相同的任务分数。反过来，一轮最终失败，也不代表过程中没有好的局部合作：团队可能完成了交接、共享了设施，甚至已经制造出目标，只是在最后检测或后续保持上出了问题。

因此，我们没有急着再设计一个“合作总分”。总分会再次把不同问题压成一个数字。我们先建立了一本可以追溯的**协作流水账**。

## 这本协作流水账记录什么

我们把聊天、命令、位置、库存、方块变化和任务进度按时间排在一起，只追踪五件普通人也能理解的事情：

1. 每个 Agent 承诺要做什么；
2. 它随后实际做了什么；
3. 世界和库存发生了什么变化；
4. 谁为谁提供了下一步所需的资源、设施或空间条件；
5. 出现异常后，团队有没有恢复。

例如，“我来完成建筑”只是一句承诺。只有当它后面出现相应命令、方块变化或资源消耗时，我们才说观察到了跟进。普通问候不算承诺，聊天多也不自动代表合作好。

同样，看到一个正确方块消失时，我们不会仅凭“谁离得最近”就认定是谁拆的。证据不够时，执行者保持未知；但我们仍可以可靠记录这个世界状态后来有没有恢复。

## 发现一：很早开始行动，不等于形成了分工

### A. 人类能理解的问题

当所有成员都说“这件事我来做”，团队究竟已经形成分工，还是只是大家同时开工？

### B. 一个具体案例

在 `trial_106` 中，Bob 先认领整张蓝图，Andy 随后明确只放自己库存里的石材。最终两人的可归因贡献约为 32:32。这里出现了实际互补，但几乎没有显式协商。[原始承诺证据](../../out/trial_106/trace/bob.trace.jsonl#L14)可以复核这一过程。

### C. 一个关键统计

在 100 轮建造实验中，有 **86 轮**出现多个 Agent 在早期同时认领整张蓝图。

### D. 为什么要统计这个数字

我们需要判断“没有明确分工”是偶发失误，还是稳定的团队工作模式。86% 表明，全任务重复认领是这些 Agent 面对共同目标时的常见起点。

另一个统计补上了行动层面的信息：有 **84 轮**后来呈现出与初始库存一致的隐式分工。持有石材的 Agent 更多放石材，持有金块和石英的 Agent 更多放对应材料。

### E. 对团队管理意味着什么

行动互补可能来自共同理解，也可能只是环境替团队做了分工。管理系统不能只看“大家做了不同的事”，还要判断角色是否被明确理解、任务边界是否重叠、换一个资源配置后分工是否仍能成立。

## 发现二：传递资源，不等于合作链已经形成

### A. 人类能理解的问题

把材料交给队友之后，资源是否真的进入了对方的下一步工作？

### B. 一个具体案例

在 `trial_201` 中，Andy 请求两根木棍，Bob 随后把木棍交给 Andy，Andy 再用它完成木镐。这里不只是“发生了交接”，而是能够看到“请求→资源流动→下游合成”的完整链条。[交接命令](../../out/trial_201/trace/bob.trace.jsonl#L11)和[接收后的行动](../../out/trial_201/trace/andy.trace.jsonl#L16)提供了直接证据。

### C. 一个关键统计

在 100 轮合成实验中，我们确认了 **66 次**成功资源交接。

### D. 为什么要统计这个数字

交接次数只能告诉我们物品移动过。为了判断资源是否真正支持了团队工作，还必须继续追踪接收者做了什么。全量数据中，只有 **37 轮**形成了“交接后由接收者执行或完成下游工作”的完整链。

### E. 对团队管理意味着什么

资源流必须和后续消费一起看。产品中的团队仪表盘不仅要显示“谁给了谁什么”，还要显示这次交接满足了哪个依赖、等待是否结束、下游任务是否启动。否则，频繁搬动物资也可能只是没有结果的忙碌。

## 发现三：发现冲突还不够，还要看能否恢复

### A. 人类能理解的问题

一部分正确成果发生了损失，这是短暂调整，还是留下了永久损伤？

### B. 一个具体案例

`trial_112` 中，一个已经正确的方块后来消失，但团队随后把同一位置恢复正确，最终仍拿到 100 分。`trial_140` 也发生了恢复，但只修回其中一部分，结束时仍有五处正确状态没有恢复。两轮都出现过异常，但恢复能力不同。

### C. 一个关键统计

100 轮建造中，我们观察到 **755 个**正确世界状态丢失序列。

### D. 为什么要统计这个数字

单独统计破坏次数，会把临时纠错、材料挪用和持久损伤混在一起。继续追踪后发现，其中 **366 个**后来恢复，另有 **389 个**直到结束仍未恢复。恢复与否才告诉我们异常是否被团队消化。

### E. 对团队管理意味着什么

一个成熟的管理系统不应只报警“发生冲突”，还应跟踪冲突持续多久、是否被发现、是否改变策略以及最终有没有修复。这决定了系统应该继续观察、重新分工，还是立即介入。

## 从任务记分牌到团队管理仪表盘

这项工作的产品价值，不是再增加一个抽象分数，而是为多智能体团队提供一种可解释的运营视图。

传统记分牌显示任务进度和最终结果。团队管理仪表盘还可以显示：哪些承诺发生重叠，在证据支持时识别实际工作由谁完成，哪些资源正在阻塞下游，哪里发生等待或重复劳动，正确成果在哪里被覆盖，以及团队是否具备自行恢复能力。

这些信息有望支持三类动作。第一类是**解释**：说明一轮任务为什么慢、为什么失败，或为什么虽然成功却成本很高。第二类是**调度**：在负载集中、资源卡住或职责重叠时建议重新分工。第三类是**干预**：当交接反复失败、等待超过阈值或正确状态长期未恢复时，向 Agent 提供更具体的上下文或更换执行者。这些动作是否有效，仍需要后续对照实验验证。

这也为企业场景提供了更合适的评估方向。软件工程、客服、供应链和机器人团队的最终结果都可能相同，但中间的责任分配、依赖履行和恢复能力，是评估系统能否可靠扩展的重要因素。Minecraft 是一个可控实验场，下一步需要验证的是这套过程视角能否迁移，而不是某一种游戏技巧。

## 我们还不能下什么结论

当前 200 轮实验主要覆盖建造和合成，尚不能证明这套表示已经完整适用于所有开放世界任务。

部分方块行为无法可靠归因到具体 Agent。因此报告对世界状态的丢失和恢复较有把握，但在证据不足时不会回答“究竟是谁做的”。

现有结果建立了过程记录和诊断基础，还没有证明自动重新分工或主动干预一定能改善团队表现。下一步需要受控实验比较不同干预策略，而不能从这批观察数据直接作因果结论。

**我们正在把多智能体评估从结果记分牌，升级为能够看见分工、依赖、冲突和恢复过程的团队管理仪表盘。**

### 附录 D：解析方法与归因限制

原始来源包括世界事件、每个 Agent 的命令与库存轨迹、得分时间线、prompt 和运行日志。解析器按真实事件时间对齐这些来源，并保留原始证据。

承诺只包括明确的未来行动声明、请求和任务目标；普通聊天不纳入。协作片段由承诺改变、资源交接、进度变化、状态丢失、求助、等待和终局等真实边界划分，而不是固定时间窗口。详细规则见[解析脚本](analyze.py)和[全量分类脚本](synthesize.py)。

方块归因要求最近姿态、距离和活动命令窗口共同支持。建造任务相关方块的平均可归因率为 48.1%；证据不足时保持 `agent=null`，不把最近 Agent 自动当作执行者。库存按约 5 秒采样，快速结束的合成轮可能缺少最后一个目标物品快照。

### 附录 E：质量审计

研究按固定种子从 200 轮中分层随机抽查 40 轮，即 20%。审计核对了承诺证据行、确认交接的命令依据、现象标签的证据位置和分类边界；40 轮均通过结构与证据检查。审计中仍保留 5 个 `unknown`，因为保留不确定性比强行分类更可靠。

200 轮均有世界事件、至少两个 Agent 轨迹、prompt 目录和运行日志，JSONL 解析错误为 0。43 轮出现 Agent 重启或重连，时间线保留了所有分段。完整审计见[40 轮审计结果](derived/semantic_audit_40_trials.jsonl)和[质量报告](derived/quality_report.jsonl)。
